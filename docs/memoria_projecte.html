1. Introducció i objectius

L'IES PORÇONS gestiona cada any l'assignació de llibres de text als alumnes a través del programa públic Banc de Llibres, que permet als centres recollir, custodiar i redistribuir els llibres entre cursos. Fins ara, el centre gestionava tot el procés manualment, amb fulls de càlcul i llistats en paper, cosa que generava errors, duplicitats i molta feina administrativa.

L'objectiu d'aquest projecte és substituir aquest procés manual per una aplicació web accessible des de qualsevol ordinador de la xarxa del centre, que permeti als professors i a l'administrador:

  • Registrar i consultar l'estat de cada exemplar de llibre.
  • Assignar lots de llibres a alumnes al inici de curs i gestionar les devolucions al final.
  • Registrar incidències (pèrdues, deterioraments) i els càrrecs econòmics associats.
  • Generar albarans en PDF que es poden imprimir i enviar per correu a les famílies.
  • Filtrar la informació per rol: els professors sols veuen els alumnes de les seves classes.
Per què una aplicació web pròpia? Es va descartar l'ús de programari genèric (Odoo, aplicatius de la Conselleria) perquè no s'adaptaven al flux de treball concret del centre, implicaven llicències de pagament o depenien d'infraestructura externa fora del control del centre. Amb una aplicació pròpia sobre XAMPP, el centre manté el control total de les dades i no depèn de serveis de tercers.

3. Arquitectura de l'aplicació

3.1 Estructura de fitxers

L'aplicació segueix un patró senzill on cada pàgina és un sol fitxer PHP, organitzat per mòdul funcional. Cada fitxer conté la lògica de negoci a la part superior (processament del formulari, consultes a la BD) i la vista HTML a la part inferior.

Evolució de l'arquitectura En una fase inicial el projecte separava cada pàgina en tres capes: un fitxer d'entrada (1 línia), un controlador i una vista. Això generava ~76 fitxers PHP per a les mateixes 23 pàgines. Es va decidir fusionar les tres capes en un sol fitxer per pàgina (seguint el patró que ja usava el dashboard index.php), reduint el total a ~30 fitxers i fent el codi molt més fàcil de llegir i mantenir.

Per què separar llibres d'exemplars? Un llibre és el títol (ex. "Matemàtiques 1r ESO, Vicens Vives"). Un exemplar és la còpia física concreta amb un codi de barres propi i un estat (bo, deteriorat, perdut). Separar-los permet saber quants exemplars hi ha de cada títol, quin n'és l'estat i qui el té en cada moment, sense duplicar la informació del títol.

3.3 Helpers: Database i Auth

Database.php

Encapsula la connexió PDO i ofereix tres mètodes estàtics que eliminen el codi repetitiu de preparació de consultes: 

// Retorna un array de files
$prestecs = Database::fetchAll("SELECT * FROM prestecs WHERE alumne_id = ?", [$id]);

// Retorna una sola fila o null
$alumne = Database::fetchOne("SELECT * FROM alumnes WHERE id = ?", [$id]);

// Executa INSERT/UPDATE/DELETE, retorna last insert id o rows affected
Database::execute("UPDATE exemplars SET disponible=1 WHERE id=?", [$id]);
Per què PDO i no mysqli? PDO suporta múltiples motors de BD (per si algun dia canvia de MariaDB a PostgreSQL), té una interfície més consistent i els prepared statements prevenen injeccions SQL de forma nativa sense cap gestió manual d'escapament.

Auth.php

Centralitza tota la lògica d'autenticació, autorització i seguretat de sessió:

  • requireLogin(): redirigeix a login si no hi ha sessió; comprova timeout de 2h.
  • requireAdmin(): limita l'accés a usuaris amb rol admin.
  • requireAccessToAlumne(): comprova si el professor logat té accés a l'alumne via la taula professor_classe.
  • csrfToken() / csrfField() / csrfCheck(): gestió completa de tokens CSRF.

4. Funcionalitats implementades

4.1 Autenticació i control d'accés per rols

L'aplicació té dos rols: admin (visió global, pot gestionar usuaris i classes) i professor (sols veu i gestiona els alumnes de les seves classes assignades).

La restricció per classes es fa a dos nivells:

  • Llistes: les consultes SQL incorporen un JOIN amb professor_classe per filtrar.
  • Accions individuals: cada controlador que rep un alumne_id crida Auth::requireAccessToAlumne(), que comprova a la BD si el professor té permís. Si no, redirigeix amb error 403.
Per què la restricció a dos nivells? No n'hi ha prou de filtrar les llistes. Si un professor malintencionat escriu manualment una URL com /prestecs/devolucio.php?alumne_id=99, sense la comprovació individual podria actuar sobre alumnes d'altres professors. La doble validació tanca aquest forat (IDOR).

4.2 Gestió d'alumnes

Els alumnes es poden donar d'alta manualment o importar massivament des d'un fitxer XML (format compatible amb el sistema ITACA de la Conselleria d'Educació). La importació crea automàticament els comptes d'usuari dels alumnes i genera un CSV amb les contrasenyes inicials per distribuir a les famílies.

4.3 Gestió de llibres i exemplars

El catàleg de llibres (titol, isbn, editorial, materia, curs) es gestiona des del menú Llibres. Per a cada títol es poden crear tants exemplars com còpies físiques hi hagi, cadascun amb un codi únic que segueix el format:

  • LOT-[CURS]-[NUM]-[MAT]-[EX] per a matèries comunes (ex: LOT-1ESO-001-MAT-03)
  • OPT-[CURS]-[MAT]-[EX] per a optatives

Els exemplars s'agrupen en lots per curs, que permeten assignar tots els llibres d'un curs a un alumne amb una sola acció.

4.4 Gestió de préstecs

Hi ha dues formes d'assignar exemplars:

  • Assignar lot: seleccionar el lot corresponent al curs de l'alumne assigna tots els exemplars disponibles del lot de cop, genera l'albarà de préstec en PDF i l'envia per correu a la família.
  • Afegir optativa: per a assignatures electives, es selecciona la matèria i el sistema cerca i assigna l'exemplar disponible d'aquell títol.

4.5 Devolució i estats de préstec

En registrar una devolució, el professor indica l'estat de cada exemplar:

EstatDescripcióEfecte al sistema
RetornatEl llibre s'ha lliurat correctamentL'exemplar queda disponible per al curs següent
PendentL'alumne no ha portat el llibre avuiEl préstec queda actiu; l'exemplar segueix vinculat a l'alumne
PerdutEl llibre no es recuperaràEs crea una incidència automàtica; l'exemplar es marca com a perdut

L'albarà de devolució PDF inclou la llista de retornats, els pendents, els perduts i els possibles càrrecs econòmics pendents de pagament.

4.6 Incidències i càrrecs econòmics

Es pot registrar una incidència manualment (deteriorament lleu, greu, pèrdua, altres) o es crea automàticament quan un exemplar es marca com a perdut en la devolució. Cada incidència pot tenir un import a pagar associat; quan la família paga, l'admin la marca com a pagada des del llistat d'incidències.

4.7 Generació de PDFs (albarans)

Tots els PDFs es generen amb la classe FPDF, una biblioteca PHP que permet dibuixar cel·les, línies i text sense dependre de cap extensió externa del servidor. La classe PdfGenerator centralitza la generació dels tres tipus d'albarà: préstec, devolució i incidència.

Cada albarà inclou:

  • Capçalera amb el nom del centre (IES PORÇONS) i la data.
  • Bloc de dades de l'alumne (nom, classe, tutor).
  • Taula d'exemplars amb codi, títol, matèria i estat.
  • Zona de firmes del professor responsable i la família.
Dificultat tècnica: encoding ISO-8859-1 FPDF no suporta UTF-8 nativament. Tots els textos s'han de convertir amb iconv('UTF-8', 'ISO-8859-1//TRANSLIT', $text) abans de passar-los a FPDF. Sense aquesta conversió, els caràcters especials catalans (à, è, ï, ç…) apareixien com a caràcters incorrectes al PDF.

4.8 Notificacions per correu electrònic

Quan es genera un albarà, si l'alumne té adreça de correu de família configurada, el PDF s'envia automàticament com a adjunt. S'utilitza PHPMailer amb SMTP autenticat sobre Microsoft 365 (port 587, STARTTLS). Tots els enviaments queden registrats a la taula enviaments_correu.

Si el servidor SMTP no és accessible (per exemple, en entorn de desenvolupament), l'error es captura silenciosament amb un bloc try/catch perquè no bloquegi el flux principal.

4.9 Importació massiva des de XML

Per evitar haver d'introduir centenars d'alumnes manualment, l'aplicació permet importar fitxers XML compatibles amb el sistema ITACA (el sistema de gestió acadèmica de la Conselleria d'Educació Valenciana). Existeix un importador per a cada entitat: alumnes, professors, classes, llibres i matèries.

El procés d'importació d'alumnes crea automàticament el compte d'usuari amb una contrasenya generada aleatòriament, la desa hashejada a la BD i escriu les credencials en clar a un CSV protegit a private/exports/ per tal que l'admin pugui distribuir-les.

4.10 Historial d'auditoria

Cada acció rellevant (préstec, devolució, incidència, enviament de correu) queda registrada a la taula historial amb la data, l'usuari que l'ha fet i un text descriptiu. Això permet reconstruir tot el historial d'un alumne o d'un exemplar en cas de disputa o error.

5. Refactorització: simplificació de fitxers

A mig projecte, l'estructura de fitxers seguia el patró MVC estricte: un fitxer d'entrada, un controlador i una vista per cada pàgina, és a dir, 3 fitxers per pàgina × 23 pàgines = ~69 fitxers, més el layout i els helpers. Això suposava un total d'uns 76 fitxers PHP.

Situació inicialSituació final
alumnes/fitxa.php (1 línia)
→ src/controllers/alumnes_fitxa.php (lògica)
→ src/views/alumnes_fitxa.php (HTML) 		alumnes/fitxa.php
(lògica PHP al capdamunt + HTML a baix)

El resultat de la refactorització:

  • Eliminats 46 fitxers (23 controladors + 23 vistes).
  • Cada pàgina = 1 sol fitxer, com ja feia el dashboard index.php.
  • Navegar el codi és molt més ràpid: no cal seguir el fil de 3 fitxers per entendre una pàgina.
  • S'utilitza un script Python per fer la fusió de forma automatitzada i fiable (actualització de paths, eliminació de la línia include vista, tractament del context PHP obert).
Per què fusionar i no mantenir la separació MVC? La separació MVC aporta valor real quan la mateixa lògica pot reutilitzar-se amb vistes diferents (per exemple, una API JSON i una vista HTML del mateix recurs). En aquest projecte, cada pàgina té exactament una vista i una font de dades, de manera que la separació en tres fitxers afegeix complexitat sense cap benefici. L'enfocament d'un fitxer per pàgina és més senzill, més llegible i és el patró que usa el dashboard des del principi.

6. Seguretat

Una vegada l'aplicació estava funcional, es va realitzar una auditoria de seguretat per identificar i corregir vulnerabilitats. A continuació es descriuen les mesures implementades i la raó de cada una.

6.1 Protecció CSRF (Cross-Site Request Forgery)

Un atac CSRF consisteix a enganyar un usuari autenticat perquè el seu navegador enviï una petició POST a l'aplicació sense que ell ho sàpiga (per exemple, visitant una pàgina web maliciosa que conté un formulari ocult apuntant al nostre servidor).

Solució implementada: token sincronitzat per sessió.

  1. En cada sessió es genera un token aleatori de 64 caràcters hexadecimals: bin2hex(random_bytes(32)).
  2. Tots els formularis POST inclouen el token en un camp ocult: <input type="hidden" name="csrf_token" value="...">.
  3. Tots els handlers POST criden Auth::csrfCheck(), que compara el token rebut amb el de la sessió usant hash_equals() (comparació en temps constant per evitar timing attacks). Si no coincideix, retorna HTTP 403.
// En Auth.php
public static function csrfCheck(): void {
    $token = trim($_POST['csrf_token'] ?? '');
    if (empty($token) || !hash_equals($_SESSION['csrf_token'] ?? '', $token)) {
        http_response_code(403);
        die('Petició no vàlida.');
    }
}

Afecta 18 fitxers amb formularis POST, automatitzat amb un script Python que insereix el camp CSRF i la crida al check de forma consistent en tots ells.

6.2 Seguretat de sessió

Diverses millores a la configuració de sessions PHP:

ParàmetreValorEfecte
cookie_httponly1La cookie no és accessible des de JavaScript → protecció XSS
use_strict_mode1No accepta IDs de sessió externs → prevenció de session fixation
cookie_samesiteStrictLa cookie no s'envia en peticions cross-site → capa addicional anti-CSRF

A més, Auth::requireLogin() comprova si han passat més de 2 hores des de l'última activitat i, si és el cas, destrueix la sessió i redirigeix al login amb un missatge explicatiu.

En el login, s'utilitza session_regenerate_id(true) just després de validar les credencials, per evitar atacs de session fixation (on un atacant pot forçar a la víctima a usar un ID de sessió conegut).

6.4 Prevenció de path traversal

Dos fitxers servien fitxers al navegador a partir d'un paràmetre GET: admin/veure_pdf.php (PDFs) i admin/download.php (CSVs de contrasenyes). Sense validació, un atacant autenticat podria fer: 

/admin/veure_pdf.php?f=../../config/config.php

i llegir les credencials de la base de dades i del correu SMTP.

Solució:

  • basename() elimina qualsevol component de directori del nom de fitxer.
  • Una expressió regular valida que el nom sigui exactament del format esperat (/^[\w\-]+\.pdf$/i i /^[\w\-]+\.csv$/i).
  • Si el fitxer no existeix o el nom no és vàlid, es retorna HTTP 404.

6.5 Mode de depuració en producció

Durant el desenvolupament, define('DEBUG', true) activava display_errors = 1, cosa que mostrava els missatges d'error de PHP directament al navegador. Un missatge d'error típic inclou la ruta absoluta dels fitxers al servidor, la consulta SQL que ha fallat, i a vegades fins i tot valors de variables.

Per a producció s'ha canviat a DEBUG = false: els errors es registren al log del servidor (/opt/lampp/logs/php_error_log) però no es mostren mai a l'usuari final.

Nota sobre les credencials a config.php El fitxer config/config.php conté la contrasenya de la base de dades i les credencials SMTP. Està exclòs del repositori Git mitjançant .gitignore per evitar que s'exposi públicament a GitHub. Existeix un fitxer config.example.php sense credencials reals per guiar la configuració en nous entorns.

8. Conclusions i millores futures

Conclusions

L'aplicació Banc de Llibres cobreix totes les necessitats identificades al plec de requisits del centre: gestió de préstecs, devolucions, incidències, generació de documents PDF, notificacions per correu i control d'accés per rols. S'ha posat especial èmfasi en la seguretat (CSRF, control d'accés per als professors) i en la senzillesa del codi (un fitxer per pàgina, sense dependències externes innecessàries).

El projecte ha permès aplicar de forma pràctica coneixements de PHP, SQL, HTML/CSS, seguretat web i gestió de projectes amb Git.
